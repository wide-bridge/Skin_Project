# 02_작업지시서_(시스템아키텍처_및_서비스기반구축)

## 1. 목적

피부질환 진단 서비스를 위한 기본 시스템 아키텍처를 확정하고, 이후 구현이 가능한 앱/서비스/데이터/라우팅 경계를 문서 기준으로 고정한다.

이번 02 단계의 목표는 아래와 같다.

- React + FastAPI 기반 서비스 구조 확정
- VLM, RAG, Vision, HITL의 모듈 경계 확정
- 오케스트레이션 흐름 확정
- 03 이후 단계에서 바로 데이터/모델/서비스 구현이 가능하도록 구조 고정

## 2. 선행 조건

01 단계는 완료 상태를 전제로 한다.

- 프로젝트 한 줄 정의 확정 완료
- In Scope / Out of Scope 확정 완료
- 기능 축 구분 완료
- 구현 우선순위 확정 완료

## 3. 전체 아키텍처

이번 단계에서 아래 아키텍처를 기준 구조로 확정한다.

```text
React Web UI
  -> FastAPI Gateway
    -> Orchestrator / Router
      -> VLM Service
      -> Vision Service
      -> RAG Service
      -> HITL Service
    -> Logging / Session / Metadata
```

## 4. 컴포넌트 정의

### 4.1 React Web

역할:

- 이미지 업로드
- 진단 결과 카드 표시
- 챗봇형 질의응답 인터페이스
- 상담 이력 보기
- HITL 요청 상태 표시

현재 단계 결정:

- React는 별도 프론트엔드 앱으로 운영
- UI는 구현 범위에 포함
- 상세 UX는 후속 단계에서 구체화

### 4.2 FastAPI Gateway

역할:

- 이미지 추론 API
- 상담 API
- 진단 설명 API
- HITL 요청 API
- 세션 및 로그 API

현재 단계 결정:

- 서비스 백엔드는 FastAPI 기준으로 통일
- API Gateway 역할과 도메인 서비스 역할을 분리

### 4.3 Orchestrator / Router

역할:

- 입력 유형 판별
- 이미지/텍스트/혼합형 요청 라우팅
- 진단 결과와 설명 체인 연결
- 상담형 질의 분기
- 실패 시 fallback 경로 제어

현재 단계 결정:

- 오케스트레이션은 별도 모듈로 둔다
- 단일 모델 직접 호출 구조로 가지 않는다

### 4.4 VLM Service

역할:

- 피부질환 이미지 기반 구조화 추론
- 질환 후보, 신뢰도, 감별 후보, HITL 필요 여부 생성
- JSON strict parsing 가능한 결과 제공

현재 단계 결정:

- VLM은 본체 모듈로 정의
- 1차 구현 우선순위는 VLM

### 4.5 Vision Service

역할:

- 필요 시 순수 비전 모델 분기 운영
- VLM 결과 검증 또는 baseline 비교
- 성능/속도 비교용 실험 경로 제공

현재 단계 결정:

- Vision은 즉시 본체는 아니지만 구조에는 포함
- fallback 또는 compare 경로로 유지

### 4.6 RAG Service

역할:

- 진단 설명용 RAG
- 일반 피부질환 상담 RAG
- 성형/미용 상담 RAG

현재 단계 결정:

- RAG는 단일 모듈이 아니라 목적별 하위 모듈로 분리
- diagnosis / derma / plastic 구분을 유지

### 4.7 HITL Service

역할:

- 사람이 확인해야 하는 케이스 표기
- 리뷰 큐 생성
- 수정 의견 기록
- 후속 데이터 피드백 연결

현재 단계 결정:

- HITL은 구현 범위에 포함
- 세부 UX는 미정이어도 서비스 모듈은 포함

## 5. 라우팅 원칙

이번 단계에서 아래 라우팅 규칙을 기준으로 확정한다.

- `이미지 전용 요청` -> VLM 우선
- `이미지 + 설명 요청` -> VLM -> Diagnosis RAG
- `텍스트 피부질환 질문` -> General Derma RAG
- `텍스트 성형 질문` -> Plastic RAG
- `불확실한 진단` -> HITL flag
- `필요 시` -> Vision branch 재확인
